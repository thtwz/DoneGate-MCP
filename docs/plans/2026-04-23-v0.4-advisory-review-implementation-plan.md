# DoneGate v0.4 Advisory Review Implementation Plan

## Goal

Ship an advisory review layer that can record architect-style or outcome-style findings and turn them into explicit follow-up tasks without blocking `done`.

## Slice 1: Design and data model

### Outcomes

- define persistent schemas for review runs and findings
- add additive advisory summary fields to task read models
- keep lifecycle and hard gate rules unchanged

### Tasks

- add models for `review_run` and `review_finding`
- add storage for advisory records
- add read helpers for open/high-severity advisory summaries
- add tests for persistence and backward-compatible loading

## Slice 2: Provider adapter layer

### Outcomes

- introduce a provider abstraction that can support manual and host-assisted review sources
- normalize provider output into a shared result shape

### Tasks

- define a provider interface
- add a manual provider
- add a host-skill provider placeholder contract
- add service-layer orchestration for running a review against a task and checkpoint
- add tests for normalized output handling

## Slice 3: CLI and MCP integration

### Outcomes

- make advisory review operable from both CLI and MCP
- expose finding inspection and disposition workflows

### Tasks

- add `task review`
- add `review list`
- add `review disposition`
- add `task create-from-finding`
- mirror these capabilities in MCP tool schemas and server adapters
- add CLI and MCP tests

## Slice 4: Trigger integration and follow-up task spawning

### Outcomes

- let `submit` and pre-`done` checkpoints produce advisory findings
- make it easy to convert findings into tracked tasks

### Tasks

- add policy/config for advisory checkpoints
- wire trigger evaluation into task progression flows
- add source linkage between findings and spawned follow-up tasks
- expose advisory summaries in dashboard and supervision
- add end-to-end tests for:
  - no findings
  - findings with proceed recommendation
  - findings with follow-up task generation

## Delivery notes

- keep v0.4 advisory only
- do not change hook blocking behavior
- prefer compact normalized summaries over verbose stored prompts
- document how host-specific skill systems can plug into the provider adapter boundary
