# DoneGate v0.3 Roadmap

## Goal

Turn DoneGate from a lightweight supervision layer into a more reliable delivery control plane for multi-branch, multi-task AI coding workflows.

## Completed Slice

### Slice 1: Branch context and repo-root normalization

Delivered in `TASK-0004`:
- branch-scoped active task resolution
- repo-root-aware normalization for relative references

## Remaining v0.3 Slices

### Slice 2: Task scope ownership and file coverage

Add optional task-owned path scopes so DoneGate can reason about whether changed files are actually covered by the active task, instead of only checking whether an active task exists.

Expected outcomes:
- tasks can declare owned file globs or path prefixes
- supervision can distinguish covered vs uncovered changes
- dashboard can surface task mismatch more precisely

### Slice 3: Richer supervision states and policy hooks

Upgrade supervision from `clean / needs_task / tracked` into more actionable statuses and use them in hooks and CLI policy decisions.

Expected outcomes:
- statuses such as `task_mismatch`, `stale_verification`, `stale_docs`, and `needs_revalidation`
- hook and CLI flows can decide whether to warn or fail based on policy
- teams can move from passive reporting to active enforcement

### Slice 4: Proactive agent/worktree enablement

Make DoneGate more self-starting for agentic workflows by improving default activation and worktree ergonomics.

Expected outcomes:
- better worktree-aware activation hints
- stronger Codex/Hermes onboarding path
- reduced manual setup and reduced chance of untracked work at session start

## Ordering Rationale

1. Branch context had to land first because it makes the current active task model trustworthy.
2. File coverage comes next because it determines whether a task actually owns the work in progress.
3. Richer supervision states and policy controls become useful only after coverage is more precise.
4. Proactive agent/worktree enablement should be built on top of the stronger policy and context model.

## Non-Goals for v0.3

- hosted control plane
- multi-user lock management
- PR or git provider API integration
- full background daemon infrastructure

## Delivery Strategy

Keep v0.3 incremental. Each slice should remain independently shippable, fully testable, and closed in DoneGate before the next slice starts.
