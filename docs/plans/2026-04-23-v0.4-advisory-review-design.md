# DoneGate v0.4 Advisory Review Design

## Goal

Extend DoneGate from a delivery compliance gate into an advisory outcome review layer that can catch "accepted but not actually useful" work before the team forgets about it.

v0.4 should stay advisory:
- review findings do not block `done`
- review findings do not override verification or doc-sync rules
- review findings can suggest follow-up tasks, deviations, or manual review

## Problem Statement

DoneGate currently answers a narrow but important question:

- was the work verified?
- was documentation synchronized?
- are required artifacts present?
- did spec drift reopen the task?

That protects delivery hygiene, but it does not protect user value.

A task can still satisfy the current gate while missing the real user need:
- the implementation passes tests but solves the wrong problem
- the acceptance path is technically complete but still awkward or incomplete for users
- the implementation closes the local task while creating obvious downstream work that no one records
- the implementation follows the literal spec while missing the real intent behind the request

The missing layer is not another hard gate. It is a structured review stage that can surface these gaps and turn them into explicit tracked work.

## Design Principles

1. Keep core delivery gates deterministic.
2. Treat advisory review as a separate evidence stream, not as a substitute for verification.
3. Make provider integration pluggable so DoneGate does not hardcode a single host-specific skill system.
4. Make findings actionable by allowing follow-up task generation and explicit disposition.
5. Default to low noise. Review should run at high-value checkpoints, not on every state change.

## Why Advisory First

Advisory mode is the right first release because outcome quality is partly subjective.

If DoneGate turns architecture or product judgment into a hard gate too early, the system will:
- create false positives
- slow down small changes
- encourage bypass behavior
- make host-specific review quality part of the core lifecycle contract

v0.4 should first prove that the review layer can produce useful findings and convert them into tracked work. Only later should the project decide whether any finding classes deserve policy enforcement.

## Conceptual Model

Add a new review layer alongside verification and doc sync.

### New concepts

- `review_run`
  A structured execution of a reviewer at a particular checkpoint for a task.

- `review_finding`
  A discrete issue, concern, or opportunity emitted by a review run.

- `review_provider`
  The mechanism that performs a review. In v0.4 this can be manual or host-assisted.

- `finding_disposition`
  The decision taken on a finding:
  - `open`
  - `accepted`
  - `spawned_followup`
  - `waived`
  - `resolved`

### Review dimensions

The first release should focus on one dimension:
- `outcome_gap`

This dimension asks whether the current implementation is likely to miss the underlying user need even if it passes its formal task acceptance checks.

Future dimensions can include:
- `architecture_risk`
- `workflow_gap`
- `ux_gap`
- `integration_gap`

## Where Reviews Trigger

v0.4 should trigger advisory review at two checkpoints:

1. `task submit`
   This is the right moment to ask whether the implementation meaningfully covers the intended use case before the team treats it as ready for verification.

2. pre-`task done`
   This is the right moment to ask whether the task is truly complete in user terms, not just compliant in process terms.

These triggers should be optional and policy-driven.

Recommended default:
- run advisory review on `submit`
- run advisory review on `done` if there are code changes or new verification/doc evidence since the previous review

## Provider Architecture

DoneGate should not embed a Codex skill directly into the core service layer.

Instead, define a provider interface like:

- `provider_id`
- `mode`
- `supports_auto_followups`
- `run(task, project, checkpoint, context) -> review_result`

v0.4 providers:
- `manual`
  Records findings from a human or external orchestrator.
- `host_skill`
  Uses the current host's review capability, such as an architecture-oriented skill or prompt template.

The important boundary is that DoneGate stores normalized review results, not provider-specific raw prompts as its main contract.

## Review Result Shape

A normalized advisory result should include:

- `review_run_id`
- `task_id`
- `checkpoint`
- `provider_id`
- `summary`
- `overall_recommendation`
  - `proceed`
  - `proceed_with_followups`
  - `needs_human_attention`
- `findings`

Each finding should include:
- `finding_id`
- `dimension`
- `severity`
  - `low`
  - `medium`
  - `high`
- `title`
- `details`
- `recommended_action`
- `suggested_task_title`
- `suggested_task_summary`
- `suggested_owned_paths`
- `status`

## Follow-up Task Generation

This is the highest-value part of the feature.

If a review detects that a task is formally acceptable but still incomplete in user terms, DoneGate should support:

1. storing the finding
2. marking it as `open`
3. generating a follow-up task from the finding
4. linking the follow-up task back to:
   - the source task
   - the source review run
   - the source finding

That turns "architectural concern" from conversation into tracked work.

## State Model Changes

Add new persisted state:

- `review-runs/`
- `review-findings/`
- optionally a lightweight `advisories.json` read model

Task payloads should gain summary fields such as:
- `last_reviewed_at`
- `open_advisory_count`
- `high_severity_advisory_count`
- `last_review_recommendation`

This should remain additive and backward compatible.

## Dashboard and Supervision Impact

Dashboard should surface advisory information without treating it as a delivery failure.

Recommended dashboard additions:
- tasks with open advisory findings
- tasks with high-severity advisory findings
- tasks with follow-up suggestions not yet spawned

Supervision should gain advisory summaries, not new blocking statuses.

Possible additions:
- `advisory_summary.open_count`
- `advisory_summary.high_severity_count`
- `advisory_summary.last_checkpoint`

These should inform humans and agents, not block hooks in v0.4.

## CLI and MCP Surface

v0.4 likely needs:

- `task review`
  Run or record an advisory review for a task and checkpoint.

- `review list`
  Show runs and findings.

- `review disposition`
  Accept, waive, resolve, or mark a finding for follow-up.

- `task create-from-finding`
  Create a follow-up task from a stored finding.

MCP should expose the same capabilities so host agents can:
- request an advisory review
- inspect findings
- spawn follow-up tasks
- record human disposition

## Recommended v0.4 Flow

1. Task moves to `awaiting_verification` via `submit`.
2. DoneGate records or requests an advisory review.
3. Review emits zero or more findings.
4. Verification and doc sync proceed as normal.
5. Before `done`, DoneGate shows open findings.
6. The operator may:
   - proceed anyway
   - spawn a follow-up task
   - waive a finding
   - record a deviation
7. Task may still reach `done`, but the advisory trail remains visible.

## Non-Goals

v0.4 should not:
- block `done` on advisory findings
- require a hosted LLM service
- encode one host's skill system as a core dependency
- promise perfect product judgment
- replace human product ownership

## Risks

### Risk: too much noise

Mitigation:
- run only at high-value checkpoints
- support severity thresholds
- prefer one compact summary plus explicit findings

### Risk: provider lock-in

Mitigation:
- keep normalized result schema provider-agnostic
- isolate provider invocation behind an adapter boundary

### Risk: advisory data becomes another ignored log

Mitigation:
- support follow-up task generation
- surface advisory counts in dashboard and task views
- require explicit disposition on findings before they disappear from attention

## Success Criteria

v0.4 is successful if DoneGate can:

1. run or record a structured advisory review at `submit` and pre-`done`
2. store findings independently from verification and doc-sync evidence
3. show advisory summaries in task and dashboard views
4. convert a finding into a follow-up task with source linkage
5. keep the existing hard gate behavior unchanged

## Recommendation

Build this as `advisory review`, not as an `architect skill hard gate`.

The first release should prove that DoneGate can help teams discover and track value gaps. If that works well, a future release can decide whether selected review dimensions should become policy-enforced.
