# DoneGate v0.2 Delivery Supervision Design

## Goal

Upgrade DoneGate from a task completion gate into a lightweight delivery supervision layer that can more reliably track AI-driven work from the moment coding starts.

## Problem

v0.1 can correctly block `done` when verification, documentation, artifacts, or spec freshness are missing. That solves the end of the workflow, but not the start and middle of the workflow.

The main gaps are:
- repositories must be manually initialized and wired
- agents can still start coding before a task exists
- hooks depend on explicit `TASK_ID` context
- the system does not proactively detect untracked work

## v0.2 Scope

### 1. Bootstrap and install flow

Add a first-class CLI entrypoint that can set up a repository for DoneGate supervision with one command.

Expected outcomes:
- initialize `.donegate-mcp`
- install bundled hook scripts into `.git/hooks`
- write minimal repo-local metadata needed by future supervision features
- reduce setup from multi-step docs to a single operator command

### 2. Active task context

Add a repository-local session/context file that lets DoneGate infer the active task for hooks and agent tooling.

Expected outcomes:
- current branch or current session can be mapped to an active task
- hooks can resolve task context without requiring `TASK_ID` every time
- agent workflows can explicitly set and inspect active task state

### 3. Untracked work detection

Add a read-model and CLI surface that flags repository work that is not currently tied to an active DoneGate task.

Expected outcomes:
- detect changed tracked files when no active task is set
- detect active task mismatch when the repo has work but the task is stale or absent
- expose a machine-friendly status for hooks, agents, and dashboards

## v0.2 Non-Goals

- full background daemon or watcher service
- multi-user coordination across many workspaces
- git provider APIs or PR-native integrations
- automatic task synthesis from arbitrary natural language

## Architecture Direction

Keep the existing shape of the system:
- domain rules in `src/donegate_mcp/domain/`
- CLI orchestration in `src/donegate_mcp/cli/main.py`
- MCP as a thin adapter over service methods
- state on disk under `.donegate-mcp/`

Add three focused layers:
- bootstrap/install service logic for repo adoption
- active-context persistence and lookup
- work-detection read model for git-based supervision signals

## Proposed State Additions

Inside `.donegate-mcp/` add:
- `session.json`: active task and branch-oriented working context
- `supervision.json`: latest untracked-work or missing-context findings

These files are derived operational state, not source of truth for task facts.

## Milestones

1. Bootstrap/install command and tests.
2. Active task context storage, CLI commands, and hook fallback logic.
3. Untracked work detection with dashboard/CLI output.
4. MCP surface expansion for supervision-oriented tooling.

## Acceptance Criteria

- A single bootstrap command can initialize a repository and install hooks.
- Hooks can resolve task context from repo-local state when `TASK_ID` is absent.
- CLI can set, show, and clear the active task context.
- CLI can report whether the repository has work that is not tied to an active task.
- Existing v0.1 lifecycle and gating behavior remain intact.

## Risks

- Git work detection can become noisy if it does not clearly separate meaningful changes from harmless state.
- Hook installation must be safe and avoid unexpectedly overwriting custom user hooks.
- Active task inference must stay deterministic and debuggable.

## Delivery Strategy

Implement v0.2 as additive supervision layers on top of the existing gate engine. Do not weaken the current lifecycle model. Improve adoption first, then context continuity, then proactive detection.
