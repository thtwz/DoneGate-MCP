# DoneGate v0.3 Branch Context and Path Normalization Design

## Goal

Improve DoneGate's supervision fidelity by making active task context branch-aware and by normalizing repository-relative path references against a stable repo root.

## Problem

v0.2 added a single active task slot and supervision reporting, but two gaps remain:

- one active task is not enough when multiple branches or worktrees are active
- relative path references can behave inconsistently across CLI, MCP, hooks, and different working directories

These issues weaken the trustworthiness of both supervision and gate enforcement.

## v0.3 Slice Scope

### 1. Branch-scoped active task context

Extend session state so DoneGate can map the current git branch to its active task while keeping a global fallback.

Expected outcomes:
- activating a task from a git repo records branch-specific context
- reading active task context prefers the current branch binding
- clearing context can remove branch-local bindings without erasing unrelated branch assignments

### 2. Stable repo-root path normalization

Introduce a canonical repository root into project state and normalize filesystem references against it.

Expected outcomes:
- `spec_ref` can be stored consistently even when created from different working directories
- `required_doc_refs` and `required_artifacts` resolve reliably during gate checks
- relative doc paths stop depending on the host process current working directory

## Non-Goals

- path ownership or task file-scope matching
- multi-repo task registry
- multi-agent locking or reservation semantics
- git provider APIs

## Architecture Direction

Persist `repo_root` in project state and `active_tasks_by_branch` in session state. Use git branch detection only when a repo root is available and the repo is valid. Keep the lifecycle engine unchanged; only normalize path-bearing inputs at service boundaries.

## Proposed State Changes

### `project.json`

Add:
- `repo_root` (nullable string)

### `session.json`

Evolve from a single task slot to:
- `active_task_id`
- `active_tasks_by_branch` (mapping of branch name to task id)
- `last_repo_root` (optional convenience field)

## Acceptance Criteria

- Activating a task inside a repo records a branch-specific binding.
- Reading active task context from that branch returns the branch-bound task.
- Clearing active task context from that branch clears only that branch binding.
- Relative `spec_ref`, `required_doc_refs`, and `required_artifacts` become stable under repo-root resolution.
- Existing v0.2 bootstrap, active task fallback, and supervision reporting stay green.

## Risks

- detached HEAD or unnamed branches require clean fallback behavior
- aggressive path normalization could accidentally rewrite non-path reference values
- old state files must continue to load safely

## Delivery Strategy

Implement branch-aware context and repo-root normalization together because both improve supervision determinism. Keep the first v0.3 slice focused and avoid pulling in task file ownership or blocking policy changes yet.
